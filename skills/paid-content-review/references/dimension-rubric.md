# 维度判定标准（dimension rubric）

每个维度给三样东西：**查什么细项**（动手时逐条核对）、**失败信号**（命中即记问题）、**增强方向**（写进报告的「怎么改」）。开审前通读，审某一维度时回查对应段落。

适用标记：`整课` 仅整门课时用 / `两者` 单篇与整课都用。单篇模式只跑标 `两者` 的维度。

---

## D1 学习路径与梯度　`整课`

**查什么**：从入门 → 配置 → 核心心智（如 Context）→ 能力封装 → 外部连接 → 自动化 → 多 Agent → 项目实战 → 团队化 → 生态边界，是否逐层推进；某主题是否出现得过早（前置没铺）或过晚（早该用上）；难度是否循序，有无断崖式跳变。

**失败信号**：开篇连续讲抽象概念、学员三章后仍不知道能做什么；高级特性夹在入门模块里；实战项目排在学员还没学会必要能力之前；相邻两章难度断层。

**增强方向**：调整章节顺序；把过早的高级内容后移、把该早用的能力前移；在难度跳变处加一节过渡/前置回顾。

---

## D2 模块分层与边界　`整课`

**查什么**：每个模块的职责是否一句话说得清；模块之间有没有职责重叠、边界模糊；同一主题是否被拆到多个模块各讲一点；单个模块内部内容是否过散（什么都沾一点、没有主线）。

**失败信号**：两个模块都在讲「配置」却没说清分工；一个概念在 A 模块开个头、B 模块又重起炉灶；某模块像「杂项收纳箱」。

**增强方向**：重新划定模块边界；把跨模块散落的同主题内容收拢到一处；给每个模块补一句明确的「本模块负责什么、不负责什么」。

---

## D3 前后依赖　`整课`

**查什么**：课程是否建立了清晰的「先学什么、后学什么、为什么这样排」；哪些章用到了前面没讲的概念（依赖未铺垫）；哪些章应当前移/后移；章节之间有没有承上启下（上一章结尾 → 下一章开头的衔接）。

**失败信号**：第 N 章默认你已会某操作，但它在第 N+3 章才讲；章与章之间硬切、无衔接句；调换两章顺序读起来反而更顺。

**高杠杆动作（必做）**：抽取每章**章首的承上启下句**（「上一章我们…」「前面几章接了…」），核对它引用的内容是否真的是它的前序章节。重构调整过模块顺序的课，这里最容易留尾巴——衔接句还停在旧顺序，引用了「按新顺序还没学到」的后置模块。这一步成本极低却最能精准抓出断层。

**增强方向**：补前置铺垫或加「前置：先看 Mx-xx」回链；调整顺序；改写过时的衔接句；在章尾/章首加一句承上启下。

---

## D4 结构完整性　`整课`

**查什么**：以 manifest/课表为基准核对**实际文件**——编号是否连续无断层；清单里有的章节文件是否都存在；有无空章（只有标题没正文）、墓碑页/废弃页；每章 `status`（如 written/published）是否与实际内容状态相符；文件名有无拼写遗留（如 cladue / claudecode 混用）。

**失败信号**：课表写到 M5-12 但文件只到 M5-09；点开某章是空的或是「本章待补」；status 标 published 实则半成品；目录里混着旧编号/拼错的文件名。

**反向核对（别把 manifest 当绝对真相）**：manifest 的备注/status 本身也可能过时。对它声称的问题要回到实际文件验证——例如备注警告「文件名有 cladue 拼写遗留」，要 grep 确认该拼写是否真的还存在；若文件早已改对，那是 manifest 备注本身陈旧（记为 D4/D9 问题）。基准错了，照基准审就会错。

**关键计数必须用脚本核，别用脑算**：总讲数、每模块章数、缺章/墓碑数这类数字会直接写进结论甚至改稿，一旦手算错（多模块求和最容易错一两位），整条发现都建立在错数上、还会误导后续动作。凡是要落进报告或据以改文件的计数，一律用脚本数实际文件，例如：
> `for d in M*/; do echo "$d $(ls "$d"/M*.md|wc -l)"; done` 再求和；墓碑页用 `grep -rl "已并入" .` 数。
脚本输出与你的预期不符时，**信脚本、复查预期**，不要反过来。涉及「品牌讲数 vs 实际讲数」（如对外『100 讲』）这种会触发改稿的判断，更要先把实际数字钉死再下结论——宁可多跑一条命令，也别让一次心算错误带偏整份审查（甚至误删作者正确的品牌话术）。

**增强方向**：补齐缺章或从课表删除；清掉/合并墓碑页；校正 status 与过时备注；统一文件命名。这是**学员最容易直接发现、最伤信任**的一类，命中基本都是 P0。

---

## D5 内容准确性与时效　`两者`

**查什么**：所有出现的 CLI 命令、参数、配置写法、API、模型名称与能力描述、文件路径、工具名/产品名，是否仍然正确、是否符合**当前版本**；标出哪些是「容易过时、上线前必须复核」的点。

**失败信号**：命令参数已改名/废弃；配置字段过时；模型能力或定价说法陈旧；路径/工具名/包名写错；照着文档一步步做会跑不通。

**横向比对（高频硬伤来源）**：同一命令/包名往往在「概览速查表」和「动手详解章」各出现一次——务必比对两处是否一致。实测中最严重的 P0 就是概览表与动手章写法自相矛盾（一边旧包、一边新包）。学员从速查表复制就会跑不通。同一事实多处出现时，**全部对齐到现行正确写法**。

**增强方向**：直接改成现行正确写法；易变内容用稳妥表述（「以你本机 CLI / 官方文档为准」），而不是堆考古说明。

### 改稿纪律：过时内容「直接换掉」，不要留给学员看考古

审查报告可以对作者写「旧包已弃用 / 需替换」——那是给创作者的内部话。**一旦进入改稿（改学员会读到的正文）**：

| 要做 | 不要做（读者噪音） |
|------|-------------------|
| 正文只呈现**当前推荐**的命令、包名、配置、工作流 | 写「已归档 / 已弃用 / 不要再安装 xxx」「旧包是…」「重要时效提醒：勿用…」 |
| 用现行路径自然讲清楚（例如 GitHub 用官方 Docker MCP；数据库用只读客户端 + Claude） | 先列一张「旧包名 → 现状 → 禁止」对照表再教新法 |
| 安全边界用正面教法（「用只读账号」「别把生产写权限交给 MCP」） | 用「因为旧实现曾有安全问题所以…」当开篇钩子 |
| 报告「上线前必复核」清单留给作者；学员文里最多一句「以本机帮助为准」 | 把「发布前复核 / 上线前动作 / 过时包名比少一个 MCP 危险」写进教程正文 |

> **一句话**：作者视角可以谈过时；**学员视角只应看到「现在怎么做」**。改时效硬伤 = 无感替换，不是加警告条。

> **纪律**：此维度**必须全量扫描**，不能抽样——「照着做跑不通 = 退款」。专项审查模式优先单独跑这一项。横向比对速查表与动手章时，两边都改成**同一套现行写法**，不要一边保留旧包、一边加弃用说明。

---

## D6 深浅与梯度　`两者`

**查什么**：该展开讲透的地方是否一笔带过；付费内容的核心溢价就是「比免费深」——是否只停在「是什么」而没讲「为什么、怎么权衡、什么时候别用」。对知识星球基础内容，深读篇还要核对是否出现至少一类可带走的增量（与 U1 对齐）：

- **可复用方法/步骤**（不是概念罗列）
- **判断标准**（何时用 / 何时别用 / 怎么选）
- **踩坑点**（真实失败路径与避坑）
- **取舍对比**（方案 A vs B 的代价）

**失败信号**：关键机制只给一句定义；给了结论不给推导/取舍；通篇科普口吻、没有作者独到的判断或经验；读完只能「知道有个词」，带不走方法或标准。

**增强方向**：在关键处补机制解释、决策理由、边界反例、取舍对比；把「告诉你有这个功能」升级成「告诉你什么时候用、踩过什么坑」；给判断清单或决策树式小结。

---

## D7 重复与冗余　`两者`

**查什么**：跨章（整课）或篇内（单篇）是否重复讲同一概念；是否为凑字数注水（同义反复、大段背景铺垫与主题无关）。

**失败信号**：同一个概念在三章里各讲一遍且没有递进；大段排比/套话；明显的篇幅填充。

**增强方向**：合并重复段落、用回链代替重述（「Context 概念见 Mx-xx」）；删注水；把重复处改成「递进」而非「重说」。

---

## D8 可操作性与交付物　`两者`

**查什么**：是否有可复现的步骤 / 可运行代码 / 截图 / 检查清单 / 练习；学完这一讲/这个模块，学员能否产出一个**明确结果**（一个配置文件、一个能跑的命令序列、一个项目雏形），而不只是「知道了一个概念」。

**失败信号**：通篇讲道理无动手环节；代码片段不完整/跑不起来；模块结束没有任何「你现在应该已经做出了 X」。

**图文占位符完成度**：留意正文里的 `[流程图]`/`[对比图]` 等占位标记。要区分两种情况——若占位注释后面**紧跟一张真实图片**（`![...](....svg)`），那是「描述注释 + 已渲染图」的正常配对，不算缺图；只有占位标记**后面没有图**才是真未补的缺口（记为 P1/P2 缺图）。别把配对好的注释误报成缺图。

**增强方向**：补可复制的步骤/代码块、关键界面截图、章末检查清单或小练习；补齐真未填的图文占位；为每个模块设计一个明确交付物。

---

## D9 一致性　`两者`

**查什么**：术语（同一概念是否全程同一叫法）、人称（你/我/我们是否统一）、写作风格、Markdown 格式（标题层级、代码块、表格样式）、编号、跨章回链格式，是否在全课/全篇统一。表达层面配合 U2：术语要准，但讲解不能绕——同一概念不要换三种黑话。

**失败信号**：同一概念前后叫法不一；人称跳动；有的章有「本章小结」有的没有；回链格式五花八门——整体读起来像多人拼接。

**增强方向**：统一术语表与人称；规范章节模板（开篇问题意识 / 概念方法 / 示例操作 / 小结下一步 / 回链）；统一格式与回链写法。拼接感 = 廉价感。

---

## D10 付费价值　`两者`

**查什么**：每一讲是否具备至少一项**超出免费科普**的价值——判断框架 / 可落地实操步骤 / 具体命令配置 / 完整工作流 / 踩坑经验 / 真实案例 / 可复用模板。逐讲判断「这一讲，免费文章给不了的是什么」。

**失败信号**：通篇能在官方文档/免费博客原样找到；只有概念罗列没有作者增量；像免费引流文充数。

**增强方向**：为「太浅/太虚/太像免费」的章节补上述七类价值中的至少一项；明确指出哪些章**不适合作为付费课的一讲**（建议合并、降级为附录或删除）。

---

## D11 定价支撑　`整课`

**查什么**：内容总厚度配不配当前定价；与对标的免费教程 / 竞品付费课相比，差异点（护城河）是什么、是否说得出口；哪些模块撑不起定价、和免费没拉开差距。

**失败信号**：定价对标某付费课，但深度只到免费水平；说不清「凭什么收这个钱」；大半内容可被一篇免费长文替代。

**增强方向**：指出需要加厚的模块；提炼并强化差异化卖点（独家工作流/实战项目/体系化）；必要时建议调整定价或重组内容结构。

---

## D12 商业吸引力与包装　`两者`

**查什么**：标题、模块名、章节顺序、毕业项目设计，是否足以支撑销售页承诺；哪些标题不够抓人、虽然正确但「不像用户愿意付费学的表达」（太平、太技术、看不出收益）。

**失败信号**：标题是功能名而非收益（「权限配置」vs「别让 AI 误删你的代码」）；模块名平淡；毕业项目撑不起「学完能做出 X」的承诺；销售页吹的点课里找不到。

**增强方向**：给出可替换的标题建议（突出收益/痛点/结果）；强化毕业项目与销售承诺的对应；让模块名更有钩子。

---

## D13 发布风险综合　`两者`

不是新维度，而是把前面发现**按风险类型重新归类**，回答「现在上线最大的雷在哪」：

- **结构风险**：编号/缺章/墓碑（来自 D4）。
- **内容风险**：准确性硬伤（来自 D5）。
- **时效风险**：易过时、上线前必复核项（来自 D5）。
- **付费感风险**：太浅/重复/像免费（来自 D6/D7/D10）。
- **完整度风险**：缺交付物/缺练习/承诺没兑现（来自 D8/D12）。
- **篇级体验风险**：深度/易懂/逻辑三硬指标未过（来自 D14 / U1–U3）——知识星球按篇消费，单篇翻车会直接伤害续费与口碑。

**产出**：每类给出「上线前必须解决」的最关键 1–3 项，汇入报告整改优先级的 P0。

---

## D14 知识星球篇级体验（U1–U3）　`两者`

对应 SKILL 中的**用户视角三硬指标**。单篇必跑；整课对每篇深读章必跑。三条**同时满足**才算过关——不是「有一条亮点就行」。

### U1 有深度、有价值

**查什么**：读者能否带走可复用增量——方法/步骤、判断标准、踩坑点、取舍对比、模板清单中至少一项；是否明显深于免费科普/官方入门文。

**失败信号**：表面介绍、「功能说明书」体；无作者判断；读完无法回答「我下次遇到同类问题怎么做」。

**增强方向**：补决策标准表、踩坑清单、最小可复用工作流；把定义段压缩，把「怎么选/怎么避坑」写厚。

### U2 专业又易懂

**查什么**：术语是否准确；讲解是否绕；是否兼顾「新手能跟」与「老手有收获」（目标读者若偏一端，另一端用注脚/进阶框即可，但不能两端都空）。

**失败信号**：术语堆砌、长句绕晕；或过度口语导致不专业；只有小白向没有进阶点；只有专家口吻没有入门梯子。

**增强方向**：术语首次出现给白话；关键概念用类比 + 正式定义；给「进阶阅读/老手注意」旁注；删掉不影响理解的黑话。

### U3 逻辑清晰

**查什么**：单篇叙事是否大致遵循 **开篇问题意识 → 概念/方法 → 示例/操作 → 小结/下一步**；小节顺序是否顺畅、有无跳戏/倒置。

**失败信号**：开篇直接堆概念无问题场景；先丢大段操作再补概念且无过渡；有理论无示例；章末无小结也无下一步；中途插入无关长文。

**增强方向**：重排小节为「问题 → 方法 → 示例 → 小结」；开篇补 2–4 句问题意识；章末补「你现在应能… / 下一篇将…」；把跑题段删掉或挪附录。

**与其他维度分工**：U1 的「够不够深」细节仍可写在 D6/D10；D14 负责**篇级体验是否同时过三关**的总判，并在报告里单独成节，避免被「付费感不足」一笔带过。
