PPT 8 · 方法篇
Skills 到底是什么：可调用的经验包
Skill 不是更长的 Prompt，而是把判断、流程、标准和资料封装成可触发的经验资产。
目录结构
my-skill/
├── SKILL.md       # 必需
├── examples/      # 可选
├── assets/        # 可选
├── references/    # 可选
└── scripts/       # 可选

一个文件夹 = 一套经验包；SKILL.md 是入口，其余按需加载。

参考案例
Superpowers requesting-code-review/SKILL.md
frontmatter 
---
name: requesting-code-review
description:
  完成任务、实现主要功能或
  合并前，用此技能验证工作
  是否符合要求
---
SKILL.md 正文（节选） 
# 请求代码审查

在问题级联扩散前，派 code-reviewer 提前发现问题。

**核心原则：** 早审查，常审查。

## 何时请求审查（必须）
- 每完成一个任务后
- 合并到 main 之前

## 反面警示（绝不要）
- 因为「很简单」就跳过审查
- 忽略 Critical 问题
和其他能力有什么区别
Prompt
一次性表达；Skill 可版本化、可复用、可自动触发。
Rules
常驻约束；Skill 是按场景调用的完整能力包。
Command
单步入口；Skill 可以带目录、附件和渐进式资料。
MCP
MCP 管「能做什么」；Skill 管「怎么想、怎么做」。
Sub-Agent
子代理是执行者；Skill 是执行者可调用的经验模块。
Plugin
插件负责分发；Plugin 内部可以包含多个 Skills。

几乎无限
自动触发
关键词匹配 description
手动触发
/skill-name
路径激活
文件类型触发（如 *.ts）
PPT 12 · 方法篇
Frontmatter 字段详解
官方共 15 个字段；核心字段决定能否被正确触发，扩展字段控制权限、模型与运行方式。
字段 必需 说明
核心（日常必写）
name 否 显示名；默认目录名；小写+连字符，≤64 字符
description 推荐 功能与触发场景；自动激活主依据，≤1024 字符
when_to_use 否 补充触发话术/示例；与 description 合计 ≤1536 字符
触发与权限
user-invocable 否 默认 true；false 时从 / 菜单隐藏
disable-model-invocation 否 默认 false；true 时仅手动 /name 触发
paths 否 路径激活，如 **/*.ts
allowed-tools 否 激活期间可用工具，最小权限配置
argument-hint 否 斜杠命令参数提示，如 [filename]
arguments 否 命名位置参数，供正文 $name 替换
运行配置 · 高级（了解即可）
model 激活期间覆盖会话模型；同 /model 取值
effort 推理深度：low / medium / high / xhigh / max
context fork 时在子代理上下文运行
agent 配合 context: fork 指定子代理类型
shell !command 块 shell：bash / powershell
hooks 技能生命周期钩子，限定在本 Skill 内生效

完整字段以 Claude Code 官方文档为准；共 15 个字段，右侧为常用组合示例。

示例 · image-to-table
---
name: image-to-table
description: 将图片中的表格内容提取为
  可编辑 markdown 表格，适用于财报、
  统计报表等高精度识别。
when_to_use: 用户上传截图/扫描件，
  或提到「图片转表格」时使用。
user-invocable: true
argument-hint: '[image-path]'
allowed-tools:
  - bash
model: default
effort: medium
paths: '**/*.png'
shell: bash
---

PPT 14 · 方法篇
最佳实践：写好名称 Name 和技能描述 Description
Name 决定能否被引用和搜索；Description 决定 AI 能否在合适场景找到并激活 Skill。
Name · 动名词形式，简洁可检索
✅ 推荐
  • processing-pdfs
  • analyzing-spreadsheets
  • writing-documentation
❌ 避免
  • 模糊词：helper · utils
  • 保留字：anthropic-* · claude-*
Description · 核心功能 + 触发词 + 排除项
✅ 好的 description
对 PRD 进行需求评审，从用户价值、可行性、完整性三个维度输出评审意见。当用户说「帮我看看这个需求」「评审一下 PRD」时使用。不适用于代码审查或 UI 设计评审。
❌ 差的 description
帮你处理各种复杂的工作任务，提升工作效率

PPT 16 · 方法篇
高质量的 Skill 检查清单
上线前对照四个维度自查，避免「命中不了」「执行出错」「测试不足」。
命中不了 执行出错 测试不足
01
可发现 · 元数据
  • Description 第三人称：做什么 + 何时触发 + 关键词
  • Name 动名词可检索，避免 helper / utils
  • 启动时只预载元数据，description 是 AI 的「入口 JD」
02
可维护 · 正文结构
  • 只写 Claude 不知道的，删掉常识性解释
  • SKILL.md ≤ 500 行，详情渐进式披露
  • 引用仅一级、术语一致、示例具体
03
可执行 · 脚本与流程
  • 脚本直接执行，不把易错操作推给 Claude 猜
  • 关键步骤：执行 → 验证 → 修复 反馈循环
  • 依赖已声明，路径用正斜杠，无常数巫术
04
可验证 · 评估与迭代
  • 评估驱动：先建 ≥ 3 个用例，再补文档
  • Haiku / Sonnet / Opus 分别测试
  • 真实任务验证，收集团队反馈持续迭代

简洁可发现 → 脚本做确定的事 → 多模型验证 → 持续迭代

PPT 17 · 应用篇
如何用 Rules + Skill 把返工从 5 轮降到 1 轮
通过 Rules + Skill 把隐性知识显性化，让 AI 从「临时猜」变成「稳定执行」。
  • 项目背景：增长平台交互优化，monorepo + amis@1.x 自研低代码混合开发框架
  • 验证目标：在「非标栈」上，AI 到底能稳定交付到什么程度
  • 需求内容：画布节点复制功能，lowcode 实现转为 procode 实现
  • 使用模型：Sonnet 4.6
  • 核心问题：非标栈 + 文档不足 → AI 不断在临时猜，产生大量返工

返工成本的本质是信息不对称
AI 编程的返工成本，本质是在为三类信息缺口买单——显式、隐式、动态约定。
1
显式信息缺失
需求文档写「新增图例」
但没说放哪、现有布局怎么调整
2
隐式知识缺失
框架的组件加载机制、构建流程、版本差异
AI 需要大量上下文才能理解
3
动态约定缺失
团队口头共识、临时决策、历史遗留问题
完全不在代码和文档中
解法：把隐性知识，变成每次对话都能拿到的显性上下文

路径 1：手动创建
手动创建是兜底能力——不管以后怎么用 Skill，你得知道它是怎么构成的。
  • frontmatter 三个字段：name、description（≤1024 字符）、allowed-tools

  • 会后材料包里有模板，复制过去改就行

description: |

05
《他的国》· 摩托车与 AI
左小龙与摩托车
韩寒《他的国》—— 情节忘了，这句话记得很清楚。
「他是那么喜欢摩托车，因为他觉得那是男人力量的延伸。」 —— 韩寒《他的国》
摩托车
力量的延伸 · 让你跑得更快、更远
替代不了骑手 —— 方向在你手里
AI
能力的延伸 · 替你搬砖、放大判断
替代不了人 —— 谁一辈子只工作、不干别的？
摩托车很好，但要有人骑着它，才知道要去哪里。

AI 很强大，但要有人驾驭它，才知道去哪。 —— Skills 把你的判断写成可驾驭的 AI 替身。

结语
AI 不会替代你
它是你能力的延伸
望远镜 = 视力的延伸  ·  汽车 / 摩托车 = 腿脚的延伸  ·  AI = 大脑的延伸 / 认知替身
专业最高形态不是「我能做到」，而是「我能让任何人做到」
自己能做到，是消费者；让他人也能做到，是创造者
招式可以拿去用，内容可以被传走——而创造它们的人，却有可能成为永恒
