# Signal-to-Optimization Framework｜从信号到优化建议

## 1. 转化链路

每条重点信号必须经过以下转化，不得只停留在摘要。

```text
1. 外部信号：发生了什么
2. 证据判断：为什么可信/不可信
3. 范式变化：旧范式是什么，新范式是什么
4. 能力变化：它带来了什么新能力
5. 用户变化：用户行为或预期如何改变
6. 机会判断：对用户目标对象意味着什么
7. 优化建议：应该怎么改
8. MVP 实验：怎么最小验证
9. 指标：怎么判断成功
10. 架构/流程影响：需要改哪些模块/流程/数据
11. 风险：哪里可能失败
12. 下一步：具体行动
```

## 2. 优化建议的最低标准

每条优化建议必须满足：

- 有明确目标用户或使用场景。
- 有对应外部证据。
- 有一个可执行 MVP。
- 有至少 2 个验收指标。
- 有风险和失败边界。
- 有复杂度判断。

不合格建议示例：

> “建议加强智能化能力。”

合格建议示例：

> “在客服自助服务流程中增加‘问题分流 + 推荐方案’步骤，对高频问题先给出可验证的解决路径和转人工条件。MVP 可先覆盖 Top 20 高频问题；指标是自助解决率、转人工率、首次解决时间和用户满意度。”

## 3. 架构影响类型

给每个建议标注架构影响：

- `workflow only`：只改流程、规则或操作步骤。
- `retrieval/indexing`：影响检索、索引、知识组织或搜索策略。
- `context layer`：影响用户、项目、团队或业务上下文。
- `tool integration`：需要调用外部工具/API/MCP。
- `data model`：需要新增实体、关系、状态或事件。
- `evaluation suite`：需要新增测试集、评分器、回归评估。
- `ui artifact`：需要新增界面、卡片、报告、工作台。
- `role split`：需要新增评审、复核、模拟或执行分工。
- `human gate`：需要人审、审批、复核、回滚机制。
- `observability`：需要日志、trace、错误归因、质量监控。

## 4. MVP 类型

常见 MVP 形式：

- Wizard-of-Oz 人工辅助验证。
- Prompt-only 原型。
- 离线批处理评估。
- 影子模式 shadow mode。
- 小流量 A/B。
- 单一客户/单一团队试点。
- 历史数据回放。
- 人工评审基准集。

## 5. 指标库

按主题选择指标：

### 质量指标

- 准确率
- 引用覆盖率
- 事实一致性
- 召回率/精确率
- 评审建议采纳率
- 漏检率/误报率

### 效率指标

- 完成时间
- 人工修改率
- 人工评审耗时
- 首次解决时间
- 工单处理时间

### 用户指标

- 使用率
- 留存
- 重复使用率
- NPS/CSAT
- 用户主动触发率

### 业务指标

- 成本下降
- 转化提升
- 缺陷减少
- 上线风险下降
- 支持成本下降

### 系统指标

- 延迟
- token 成本
- 工具调用成功率
- 回退率
- 失败恢复率
