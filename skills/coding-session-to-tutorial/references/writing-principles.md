# 实战教程写作原则

> 从三篇案例中提炼的「让读者真正学到东西」的写法要诀。

---

## 核心原则：真实 > 完美

### 1. 弯路必须保留

很多人写教程会把弯路删掉，只保留「正确答案」。这是错的。

**弯路是教程最有价值的部分**：
- 读者遇到真实问题时，也会走弯路；看到你也走过，会感到安心
- 弯路描述了「这个方向行不通的原因」，帮读者建立判断力
- 「尝试了 A，发现不对，转向 B」这个叙事弧度让文章有戏剧性

**写法**：
```markdown
### 第 N 阶段：[尝试 A]（约 15 分钟）
... 执行了操作 A ...
**结果**：仍然卡住 → **证实了这不是根因**，驱动第四阶段
```

---

### 2. 初步假设 ≠ 正确假设

案例背景里写「初步假设」，不是为了展示你很聪明，而是为了让读者跟着一起「猜错」，然后一起被纠正。

这个认知转变是整篇文章的情感核心：
```
读者进入时的认知 → 案例发生了什么 → 读者离开时的认知
```

如果你一开始就写了正确答案，读者没有这个转变过程，学不深。

---

### 3. AI 协作要写「具体怎么用」，不是「AI 帮了我」

**弱写法**：
> 我让 AI 分析了报错，AI 给出了解决方案。

**强写法**：
> 用户把终端完整输出（含第 3 行的 `File XXX is 549MB` 那行）粘给 AI，AI 立即锁定：这不是网络问题，是**大文件**——理由是推送 81 个对象却要 54MB，平均每对象 670KB，正常代码仓库是几 KB。用户自己以为是网速问题，没注意到这个数据异常。

要写清楚的三件事：
1. 用户给了 AI **什么信息**（完整报错 / 数字 / 截图描述，而不是转述）
2. AI 给出了**什么判断/命令**（具体）
3. 为什么这次 AI 协作**有效**（上下文完整 / AI 做了用户没注意到的数据分析）

---

### 4. 通用清单要能「直接用」

读者遇到同类问题会直接跳到清单。写清单的标准：

**能立即执行**：每条要么是一个决策（「如果是情况 A，做 XX」），要么是一行命令。
**不需要上下文**：清单里的每条，不用读前面的内容就能理解是什么意思。
**有分支**：真实问题有多种情况，清单里要有「情况 A / 情况 B」的分支，而不是只写一条路。

---

### 5. 方法论升维：从「这次」到「下次」

方法论总结不是复述解决步骤，而是回答：「下次遇到**类似**问题，我的思维框架是什么？」

升维方法：
- **问「为什么这个方法有效」**：不是「用了 git-filter-repo」，而是「用通用规则（>5MB）而不是单个文件，一次性解决根因」
- **提炼可迁移的思维**：「现象→线索→根因」这个框架，不只适用于 git，适用于所有技术问题排查
- **点出人机协作的规律**：「信息透明度决定 AI 的有效性」不只在这个案例成立

---

### 6. 标题和摘要要有具体卖点

**弱标题**：AI 编程实战：Git 问题解决
**强标题**：AI 编程实战：解决 Git Push 卡住问题

**弱摘要**：本文介绍了如何用 AI 解决 git push 的问题。
**强摘要**：真实案例复盘：如何用 AI 编程工具系统性地排查和解决技术难题（标注了方法论价值，而不只是单次问题）

---

## 常见错误清单

| 错误 | 为什么不好 | 怎么改 |
|------|-----------|--------|
| 只写最终解法，不写弯路 | 读者学不到判断力 | 保留每个「尝试但不对」的阶段 |
| 「AI 帮我解决了问题」 | 没有可复用的协作经验 | 写清楚给了 AI 什么、AI 怎么分析的 |
| 通用清单用叙述句 | 读者无法快速扫描执行 | 改成 checkbox + 命令块 |
| 方法论只是步骤复述 | 没有升维，读者下次遇到类似问题还是懵 | 提炼思维框架，加「为什么有效」 |
| 数字用模糊描述 | 缺乏可信度 | 「快了很多」→「从 60+ 分钟 → 3 秒」 |
| 命令散落在文字中 | 无法复制执行 | 全部放进代码块 |
