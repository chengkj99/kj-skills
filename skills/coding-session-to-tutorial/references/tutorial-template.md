# 教程结构模板

> 基于三篇真实案例（Git Push 卡住、HTTPS 故障排查、Skills 定时同步）提炼的结构范式。每个字段都有存在理由，不能随意删减。

---

## 完整结构

### 一、标题与导言

```markdown
# AI 编程实战：[问题动词短语]

> [一句话定位：这是什么类型的案例 + 读者能学到什么]
```

导言要包含：
- 案例类型（故障排查 / 工具实践 / 工作流搭建）
- 一个「可迁移」的提炼点，让读者知道看完有什么收获

**示例**：
> 真实案例复盘：如何用 AI 编程工具系统性地排查和解决技术难题

---

### 二、案例背景

**必须包含**：
1. **问题现象**：越具体越好。有报错信息写报错，有数字写数字，有频率写频率。
2. **初步假设**：当时以为是什么原因（哪怕是错的）

**为什么要写初步假设**：这是「认知转变」的起点。读者会产生共鸣——「我也会这么想」——然后跟着一起被打脸，这才是学习发生的时刻。

```markdown
## 案例背景

### 问题现象
- **症状**：[具体表现，能复现]
- **频率/规模**：[发生了几次，影响范围]
- **错误信息**：（如有，直接贴）

### 初步假设
[当时以为是什么，哪怕是错的]
```

---

### 三、排查/实现过程（多阶段）

**这是教程的核心**，必须按阶段展开，每阶段标注耗时。

每个阶段的固定结构：
```markdown
### 第 N 阶段：[阶段名]（约 X 分钟）

[做了什么，用了什么工具/命令]

#### AI 协作方式（重要！）
[这个阶段 AI 具体起了什么作用：提供命令 / 分析输出 / 推荐工具 / 指出遗漏]

**关键发现**：[这个阶段的结论，驱动下一步行动]
```

**阶段划分原则**：
- 每阶段有一个独立的目标和结论
- 阶段之间有逻辑驱动（上一阶段的结论导致下一阶段的行动）
- 最好有「弯路阶段」——尝试了不对的方向，这才真实

**AI 协作方式必须写清楚**，这是与普通技术博客的差异化。要说明：
- 用户给 AI 提供了什么（完整报错 / 代码片段 / 截图描述）
- AI 给出了什么（命令 / 分析 / 工具推荐）
- 为什么 AI 能帮到这里（上下文完整 / 并行检查 / 知识迁移）

---

### 四、AI 编程方法论总结

**从这个案例提炼出的可迁移方法**，不是总结这次解决步骤，而是升维到「下次遇到类似问题可以用的思路」。

必须包含以下视角中的 2-3 个：

```markdown
## AI 编程方法论总结

### 1. [思维层面]：[方法名]
[一句话描述这个思维方式，配一个具体例子说明在本案例中怎么体现的]

### 2. [工具层面]：[工具使用原则]
| 阶段 | 工具 | 用途 |
| 诊断 | xxx | xxx |
| 修复 | xxx | xxx |

### 3. [协作层面]：[人机协作原则]
[什么信息给 AI、怎么给、AI 擅长什么不擅长什么]
```

---

### 五、可复用的通用清单

**这是最重要的一节**，读者下次遇到类似问题直接翻这里。

格式要求：
- 用 `[ ]` checkbox 格式，读者可以打勾
- 每个 checkbox 对应一个决策/检查/命令，能独立执行
- 按「诊断→修复→预防」三段组织，不要全平铺

```markdown
## [问题类型] 通用排查清单

#### ✅ 第一步：[诊断类操作]
- [ ] [具体检查项]（附命令）
- [ ] [具体检查项]

#### ✅ 第二步：[定位类操作]
- [ ] [具体检查项]

#### ✅ 第三步：[修复类操作，按情况分支]
- **情况 A**：[...]
  ```bash
  [命令]
  ```
- **情况 B**：[...]

#### ✅ 第四步：[验证和预防]
- [ ] [验证步骤]
- [ ] [预防措施]
```

1. [...]

1. [...]
```

### 七、元数据尾注

```markdown
---
> **案例时间**：[YYYY-MM-DD]  
> **耗时**：[总耗时，含弯路]  
> **最终方案**：[一句话]  
> **关键突破点**：[从错误假设到正确根因的转变]
```

---

## 长度与深度标准

- **最短可发布**：背景 + 过程(2阶段以上) + 清单 = 约 1500 字
- **完整版**：全部 7 节 = 约 3000-4000 字
- **命令块**：所有命令必须放在代码块里，不能散落在文字中
- **数据具体化**：能写数字就写数字（时间、文件大小、次数）
