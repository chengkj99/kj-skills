# Weekly Digest（周报生成器）

从 git 历史自动生成结构化周报，减少手工汇总的重复劳动。

## 语言规则

所有面向用户的输出必须使用**中文**。例外：代码块内的命令、路径、变量名保持原语言。

---

## 用户口令速查

| 口令 | 作用 |
|------|------|
| **/weekly-digest** | 生成本项目本周 + 上周对比周报（默认） |
| **周报** / **本周总结** / **生成周报** | 同上 |
| **多项目周报** / **团队周报** | 聚合多个项目输出周报 |
| **只看本周** | 仅输出本周，不含上周对比 |
| **周报 2026-05-19** | 指定周起始日期 |

---

## 核心概念

### 周定义

- **本周**：从本周一到今天（含今天）
- **上周**：从上周一到上周日
- 用户可指定任意起始日期覆盖默认范围

### 两种模式

1. **单项目模式**（默认）：当前 git 仓库
2. **多项目模式**：用户提供一组仓库路径，或从配置文件读取项目列表

---

## 执行流程

### Step 0：确定模式与范围

1. 判断用户意图：
   - 包含「多项目」「团队」「跨项目」等关键词 → 多项目模式
   - 否则 → 单项目模式（当前工作目录）
2. 判断时间范围：
   - 用户指定了日期 → 以该日所在周为「本周」
   - 否则 → 自动计算本周一和上周一
3. 多项目模式下，确定项目列表来源：
   - 用户显式列出了路径 → 使用用户列表
   - 否则 → 读取配置文件 `~/.claude/weekly-digest-projects.json`（见 references/projects-config.md）

### Step 1：采集 git 数据

对每个项目执行以下采集脚本：

```bash
# 计算日期
THIS_MONDAY=$(date -v-monday +%Y-%m-%d 2>/dev/null || date -d "last monday" +%Y-%m-%d)
LAST_MONDAY=$(date -v-7d -v-monday +%Y-%m-%d 2>/dev/null || date -d "last monday -7 days" +%Y-%m-%d)
LAST_SUNDAY=$(date -v-1d -v-sunday +%Y-%m-%d 2>/dev/null || date -d "last sunday" +%Y-%m-%d)

# 本周提交
git log --oneline --since="$THIS_MONDAY" --until="$(date -v+1d +%Y-%m-%d)" \
  --format="%h %ad %s" --date=short

# 上周提交
git log --oneline --since="$LAST_MONDAY" --until="$THIS_MONDAY" \
  --format="%h %ad %s" --date=short

# 代码统计（本周）
git log --since="$THIS_MONDAY" --until="$(date -v+1d +%Y-%m-%d)" \
  --shortstat --format="" | awk '/files? changed/ {f+=$1; i+=$4; d+=$6} END {print "files:"f" insertions:"i" deletions:"d}'

# 代码统计（上周）
git log --since="$LAST_MONDAY" --until="$THIS_MONDAY" \
  --shortstat --format="" | awk '/files? changed/ {f+=$1; i+=$4; d+=$6} END {print "files:"f" insertions:"i" deletions:"d}'
```

> macOS 与 Linux 的 date 命令语法不同，脚本需兼容两种平台。详见 `scripts/collect.sh`。

### Step 2：分类与摘要

这是周报质量的关键步骤。对提交列表做智能分类，而不是机械罗列。

**分类规则**（按优先级）：

1. **按 scope 分组**：从 conventional commits 的 `scope` 字段提取（如 `feat(auth): ...` → auth）
2. **按关键词归类**：扫描 commit message 中的关键词，归入预设类别：
   - `feat` / `新增` / `实现` → 新功能
   - `fix` / `修复` / `bug` → Bug 修复
   - `docs` / `文档` / `README` → 文档
   - `refactor` / `重构` / `调整` → 重构
   - `test` / `测试` / `eval` → 测试
   - `chore` / `维护` / `CI` / `build` → 维护/基础设施
3. **合并微小提交**：同一主题的多个小提交（如连续 fix）合并为一条
4. **提取亮点**：标记影响面大或用户可见的变更（新增行数 > 200、涉及核心 API、新模块等）

**摘要撰写**：每条不简单复述 commit message，而是提炼：
- 做了什么（What）
- 为什么重要 / 影响范围（Why it matters）
- 关联的模块/文件（Where）

- {亮点条目1}：{摘要}（{关联文件/模块}）
- {条目2}：{摘要}

#### 二、{分类2标题}

#### 一、{分类1标题}

...

---

| 指标 | 上周 | 本周 | 变化 |

### Step 4：可选增强

根据用户偏好，可追加以下内容（默认不输出，用户口令触发）：

- **贡献者维度**：按 author 分组统计（口令：「按人统计」）
- **PR/Issue 关联**：通过 `gh` CLI 关联 GitHub PR 和 Issue（口令：「关联 PR」）
- **导出文件**：将周报写入文件（口令：「导出周报」→ 写入 `docs/weekly-digest/{week}.md`）

## 多项目配置

多项目模式需要项目列表。来源优先级：

1. 用户显式指定的路径列表
2. 配置文件 `~/.claude/weekly-digest-projects.json`

配置文件格式：

```json
{
  "projects": [
    {
      "name": "信贷前端 Wiki",
      "path": "/Users/didi/ai_space/loanfe-llm-wiki",
      "tags": ["knowledge", "infra"]
    },
    {
      "name": "信贷 H5",
      "path": "/Users/didi/ai_space/loanfe-h5",
      "tags": ["business", "h5"]
    }
  ],
  "defaults": {
    "include_last_week": true,
    "max_commits_per_project": 50
  }
}
```

详见 `references/projects-config.md`。

## 边界情况处理

| 情况 | 处理方式 |
|------|----------|
| 当前目录不是 git 仓库 | 提示用户切换到 git 仓库，或使用多项目模式指定路径 |
| 本周无提交 | 输出「本周无提交」，仍显示上周摘要和趋势 |
| 合并提交（merge commit） | 默认过滤 merge commit，用户可通过「含 merge」口令包含 |
| commit message 不规范 | 做最佳努力分类，归入「其他」类别 |
| 跨时区 | 使用本地时区计算周一 |
| 多分支 | 默认统计当前分支，用户可通过「全分支」口令统计所有分支 |

---

## 质量标准

一份好的周报应该：

1. **有重点**：不是所有 commit 的等权罗列，而是突出最重要的 3-5 项
2. **有上下文**：每条摘要说明「为什么重要」，而不只是「做了什么」
3. **有对比**：本周 vs 上周的趋势让读者感知节奏
4. **有结构**：按主题分类而非按时间排序，降低阅读负担
5. **简洁**：总长度控制在 100 行以内（单项目），200 行以内（多项目）

## 交付前自检

- [ ] 确认 `scripts/collect.sh`（或 Step 1 的 git log 命令）执行成功且退出码为 0，提交数、+行/-行、文件数均来自真实 git 输出，未做估算。
- [ ] 对照 `assets/templates/` 模板核验结构：单项目周报含「本周」「上周」「📈 趋势对比」三节；多项目周报含「项目概览」表和「📈 团队整体趋势」表。
- [ ] 检查提交已按主题分类（新功能 / Bug 修复 / 文档 / 重构 / 测试 / 维护），而非按时间等权罗列；每条摘要写明「为什么重要」。
- [ ] 核对「趋势对比」表中提交数、新增行、文件变更的变化百分比与上文本周/上周统计数字一致。
- [ ] 确认总长度达标：单项目 ≤ 100 行，多项目 ≤ 200 行。
- [ ] 确认 merge commit 已按默认过滤（用户口令「含 merge」时除外）；本周无提交时已输出「本周无提交」并保留上周摘要与趋势。
