# 受众视图

所有视图都必须基于同一份 Git 事实底稿。用户补充的里程碑、风险、计划或业务指标可进入相应字段，但要与 Git 事实区分，不得混写。

## 直属领导

适合“给领导”“管理者视图”。强调结果、优先级和需要决策的事项，不逐条列提交。

```markdown
## {项目名} 周报｜直属领导版

**周期：** {周一} – {周日}  
**结论：** {基于本周主题的 1–2 句总结}

### 关键产出

- **{产出}**：{做了什么；为何重要}。

### 进展与关注项

| 事项 | 当前证据 | 关注/需要支持 |
|---|---|---|
| {主题} | {Git 可验证的实现或变更} | {用户提供的风险/支持；没有则“待补充”} |

### 下周重点

- {用户提供的计划；没有则“待补充”}
```

## 团队协作

适合“团队同步”“给团队”。按模块展示变化和协作关注项；不可根据提交量评价个人。

```markdown
## {项目名} 研发周报｜团队同步

**周期：** {周一} – {周日} ｜ {提交数} 次提交

### 本周模块进展

- **{模块}**：{合并后的工作摘要；影响范围}。

### 协作关注项

- {用户提供的依赖、评审或联调事项；没有则“暂未提供”}

### 下周协作重点

- {用户提供的计划；没有则“待补充”}
```

## 高管摘要

适合“高管摘要”“给高管”。不展示 Git 统计、commit、路径或作者；最多约 15 行。

```markdown
## {项目名} 周度高管摘要

**周期：** {周一} – {周日}  
**状态：** {有明确里程碑/风险证据时才写“正常/关注”；否则“待确认”}

### TL;DR

{2–3 句：本周最重要的可验证产出及其意义。}

### 关键产出

- {业务可读的产出描述，不含内部实现细节}

### 风险 / 需要决策

- {用户提供的风险或决策；没有则“待确认”}

### 下一周期重点

- {用户提供的计划；没有则“待确认”}
```

## 客户进展

适合“客户周报”“给客户”。采用对外交付语气，但不能将代码提交等同于交付、上线或验收。

```markdown
## {项目名} 周度进展更新

**周期：** {周一} – {周日}

您好，以下是本周可确认的项目进展：

### 本周完成的工作

- {将 Git 事实转为非技术化、可对外表述的工作描述；不承诺上线或验收}

### 进行中的事项

- {仅用户确认后填写；否则“待确认”}

### 需要您确认 / 支持

- {仅用户确认后填写；否则“本周无已确认事项”}

### 下一步

- {仅用户确认的计划；否则“待确认”}
```

结尾可添加“如需进一步了解某项进展，欢迎联系我们。”不添加未经确认的交付日期、完成百分比或承诺。
