# AI 编程内容选题策略

把一条输入（动态/问题/痛点）转化为「有人看、能写出来、符合定位」的内容角度。

---

## 执行流程（每次必走）

### 第 1 步：判断输入类型

先读输入，判断是哪一类：

| 类型 | 特征 | 例子 |
|------|------|------|
| **工具动态** | 新版本发布、新功能、行业新闻 | 「Claude Code 支持了 xxx」 |
| **实战记录** | 具体问题 + 解决过程，有命令/报错/截图 | 「我今天用 Cursor 遇到了 xxx」 |
| **痛点/困境** | 程序员的模糊感受或具体卡点 | 「感觉用 AI 没有提效」 |
| **想法/灵感** | 一句话观点、对比、反常识判断 | 「我发现 Cursor 用户和 Claude Code 用户思维方式不同」 |
| **已有素材** | 旧文章、笔记、草稿 | 提供一段已有内容需要改编 |

> 如果输入是**实战记录**（有完整的问题-排查-解决流程），直接建议用 `coding-session-to-tutorial` 技能先结构化，再回来选角度。

---

### 第 2 步：如果在 LLM Wiki 中做选题，先做连载去重

当工作目录是 `kj-llm-wiki`，或用户要求“基于已发布内容、连载、后续选题、内容规划”时，必须先读：

- `wiki/playbooks/topic-bank/series-planning.md`
- `wiki/playbooks/topic-bank/index.md`

并检查已发布与待写内容：

- `raw/studio/transcripts/`
- `raw/studio/funnel/formatted/`
- `raw/studio/funnel/backlog/manual/`
- `raw/studio/funnel/backlog/queue.md`

再从以下目录挖掘知识库支撑：

- `raw/notes/`
- `raw/papers/`
- `raw/assets/`
- `raw/studio/curriculum/lessons/`
- `raw/articles/`
- `docs/`

候选选题优先作为已发布内容的**续集、方法篇、案例篇、升级篇、反面篇**，不要换标题复述旧观点。

每个候选选题必须补充 5 个字段：

- **已发布连接**：连接哪条已发布内容，避免凭空立题
- **重复风险**：是否和已发布内容重复，重复点是什么
- **连载位置**：续集 / 方法篇 / 案例篇 / 升级篇 / 反面篇
- **知识库支撑**：哪些目录或文件可作为内容池
- **下一集钩子**：这一条发布后，下一条自然接什么

### 第 3 步：对照角度模式库，找 3-5 个候选角度

引用 `references/angle-patterns.md`，逐一过以下 7 个模式，找最匹配的：

1. **认知反转型** — 输入里有被推翻的假设或反常识事实？
2. **痛点具象化型** — 能精准钉住某类程序员的真实感受？
3. **方法论提炼型** — 能从具体案例提炼出可迁移框架？
4. **工具对比评测型** — 涉及两个及以上工具的选择？
5. **实战案例记录型** — 有完整的「问题→过程→结果」？
6. **趋势解读型** — 新动态发布，需要帮程序员翻译意义？
7. **「我踩过的坑」型** — 有教训、弯路、认知升级的经历？

每个候选角度写清楚：
- **一句话描述这个角度**：读者看到后会想什么
- **核心洞见句**：这篇文章要让读者「哦，原来如此」的那一句
- **3 个候选标题**（痛点型 / 反常识型 / 方法型各一个，参考 `references/content-criteria.md`）

---

### 第 4 步：用五问框架评估，筛出最优

引用 `references/content-criteria.md` 里的五问框架，对每个角度逐项打分（✅/⚠️/❌）：

```
[ ] 符合定位（Claude Code / Cursor / Codex / AI编程方法论 / 程序员破局）
[ ] 读者真实痛点
[ ] 我有独特视角（真实经历/亲踩的坑）
[ ] 可多渠道转化（长文→短视频→小红书）
[ ] 能与已有内容连接
```

通过 3 个以上的角度标记为 ⭐ 推荐。

---

### 第 5 步：输出结构化选题报告

按以下格式输出（不要省略任何字段）：

---

**输入类型**：[工具动态 / 实战记录 / 痛点 / 想法 / 已有素材]

---

#### ⭐ 推荐角度 1：[角度名称]

**角度类型**：[认知反转型 / 痛点具象化型 / ...]

**已发布连接**：[如果适用，连接哪条已发布内容；如果是新系列，说明“新系列”]

**重复风险**：[是否和已发布/待写内容重复，以及如何避开]

**连载位置**：[续集 / 方法篇 / 案例篇 / 升级篇 / 反面篇 / 新系列]

**知识库支撑**：[可引用的目录或文件]

**下一集钩子**：[这一条发布后，下一条自然接什么]

**核心洞见**：[一句话，读者看完会「哦，原来如此」的那句]

**候选标题**：
- 痛点型：[...]
- 反常识型：[...]
- 方法型：[...]

**五问评估**：✅✅✅✅⚠️（通过 4/5）

**推荐工作流**：[content-creator 工作流 X] + 理由（一句话）

**写作提示**：[帮助启动这个角度的 1-2 个具体切入点，比如「开场可以从 xxx 场景开始」]

---

#### 角度 2：[角度名称]
（同上结构，但非推荐角度可以略写）

---

**总结建议**：
- 如果今天只写一篇：选角度 X，理由是 [...]
- 如果要做成系列：可以 [角度 1 + 角度 3] 组合，前者破认知，后者给方法

## 领域知识库（写 AI 编程内容必知）

### 核心工具定位（不要搞混）

| 工具 | 定位 | 核心用户 | 差异化 |
|------|------|---------|--------|
| **Claude Code** | CLI Agent，接管整个终端 | 工程师，喜欢控制权 | 可以跑长任务、管文件系统、自主决策 |
| **Cursor** | IDE 插件，在编辑器里对话 | 日常开发为主 | 与代码高度集成，上下文感知强 |
| **Codex / OpenAI** | API + 代理能力 | 开发者集成场景 | 适合嵌入到自己的工具里 |

### 程序员读者的核心关切（按频率排序）

1. 能不能真的提效（不是说说而已）
2. 会不会被替代（焦虑但不想承认）
3. 怎么用才是正确的（学了不会用）
4. 跟别人用的差距在哪（暗暗比较）
5. 值不值得花时间学（精力有限）

### 高传播力的内容特征

- **有真实数据**：「3 秒」「从 60 分钟到 5 分钟」「第 3 次才成功」
- **有报错/截图**：证明是真实经历，不是 AI 编的
- **有认知转变**：「我以为是 A，结果是 B」这个弧度是核心
- **有可执行清单**：读者可以今天就试的 1 件事
