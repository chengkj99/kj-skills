# remove-ai-tone-guide：如何写出有「人味」的内容

## 核心原则：真实 > 完美

**AI 生成的内容往往过于「完美」，缺乏真实感。我们的目标是：让读者感觉是一个真实的人在分享经验，而不是 AI 在输出模板。**

---

## 一、结构层面：打破规整，增加变化

### ❌ AI味表现
- 标题层级过于规整（一、二、三、四...）
- 每个章节都有固定格式：核心问题 → 关键要素 → 避坑指南
- 小标题编号过于系统化（2.1, 2.2, 2.3...）
- 所有章节长度一致，缺乏节奏感

### ✅ 人味写法
- **打破固定格式**：有些章节用故事，有些用对话，有些用案例
- **标题多样化**：
  - **禁止使用编号**：「一、二、三」→ 用描述性标题（如「第一次踩坑：我只说了一句『添加一个按钮』」）
  - 混用风格：有些用问句，有些用陈述，有些用感叹
  - 用故事化标题替代编号（如「第一次踩坑」「第二次踩坑」）
- **结构灵活**：不是每个章节都要「核心原则」「底层逻辑」「为什么有效」
- **节奏变化**：有些章节可以更短（2-3段），有些可以更长（5-6段），制造节奏感

**示例对比**：
```
❌ AI味：
### 3.1 第一层：需求层（What）
**核心问题**：你要实现什么？
**关键要素**：
**避坑指南**：

✅ 人味：
### 需求层：说清楚要什么
我刚开始也犯过这个错。有次让 Cursor 加个按钮，结果它给我生成了 5 个版本，每个都不一样。

后来我才明白，是我没说清楚位置、样式、功能。
```

## 二、表达层面：口语化 > 书面化

### ❌ AI味表现
- 高频使用「信息论视角」「底层逻辑」「核心原则」等术语
- 每段都有总结句，显得套路
- 语言过于完美，没有口语化表达
- **句式重复严重**：大量使用「我刚开始也...」「后来我才明白...」「后来我学会了...」这种固定句式
- 每个章节都有「后来我明白了/学会了」的总结句

### ✅ 人味写法
- **用口语化表达**：
  - 「信息论视角」→ 「说白了就是...」
  - 「底层逻辑」→ 「其实道理很简单...」
  - 「核心原则」→ 「我总结下来就一句话...」
  - 「提供具体修改点，而不是模糊的描述」→ 「直接说清楚要改哪里，别让AI猜」
  - 「在开始之前，先明确代码规范」→ 「先跟AI说好代码规范，别让它自由发挥」
- **避免句式重复**：
  - 「后来我才明白...」→ 「试了3次才发现...」「吃了亏之后...」「踩过这个坑，我才知道...」
  - 「后来我学会了...」→ 「试了几次发现...」「当时我就懵了...」
  - **同一句式最多出现1次**，不要每个章节都用
- **允许不完美的句子**：
  - 偶尔用短句
  - 偶尔用口语化的连接词（「其实」「说白了」「说白了就是」）
- **减少套路化总结**：不是每段都要总结，让读者自己思考

**示例对比**：
```
❌ AI味：
**核心原则**：**一次性提供完整上下文**。包括功能目标、交互细节、样式要求、技术约束。

✅ 人味：
说白了，就是一次性把话说清楚。

我刚开始也犯过这个错。有次让 Cursor 加个按钮，结果它给我生成了 5 个版本，每个都不一样。后来我才明白，是我没说清楚位置、样式、功能。
```

## 三、细节层面：具体 > 抽象

### ❌ AI味表现
- 「昨天在技术群里」缺少细节（哪个群？具体时间？）
- 例子都是通用场景，缺少个人色彩
- 没有真实对话截图或具体数据
- **案例缺乏细节**：
  - 「有次让Cursor加个按钮，我只说了一句...」- 太简单
  - 「还有一次，我想让AI优化一段代码...」- 缺乏具体场景
  - 「这是我之前遇到的一个bug」- 没有前因后果

### ✅ 人味写法
- **补充具体细节**：
  - 「昨天在技术群里」→ 「昨天在『AI编程实战群』，@老王 发了一张截图...」
  - 「有个朋友抱怨」→ 「@老王 在群里说：『我用 Cursor 写个按钮，来回改了 5 遍...』」
- **使用真实数据**：
  - 「效率提升 10 倍」→ 「从 2 小时缩短到 15 分钟」
  - 「很多人」→ 「我观察了 20+ 个程序员」
- **加入个人色彩**：
  - 具体的项目名称
  - 具体的时间、地点
  - 具体的对话内容
- **增加案例细节**：
  - 增加具体场景：什么项目、什么时间、什么情况
  - 增加情绪：当时的心情、遇到的困难（「我当时就火了」「试了3次才搞定」）
  - 增加细节：具体代码、具体错误信息

**示例对比**：
```
❌ AI味：
昨天在技术群里，有个朋友抱怨：「我用 Cursor 写个按钮，来回改了 5 遍，还不如自己手写快。」

✅ 人味：
昨天在『AI编程实战群』，@老王 发了一张截图，是他和 Cursor 的对话记录。

他说：「我用 Cursor 写个按钮，来回改了 5 遍，还不如自己手写快。这 AI 是不是有问题？」

我回了一句：「你肯定只说了一句『添加一个按钮』吧？」

他沉默了。
```

## 四、金句层面：自然 > 刻意

### ❌ AI味表现
- **金句密度过高**：3句金句/250行，显得不自然
- 同一金句重复出现
- 句式过于工整，缺乏变化
- **金句单独成段**：显得突兀，和前文连接不自然
- 金句过于「完美」，缺乏真实感

### ✅ 人味写法
- **降低金句密度**：
  - **建议全文金句数量 ≤ 2句**
  - 不是每段都要金句，让金句自然出现
- **避免重复**：同一金句最多出现 1 次
- **增加变化**：有些金句工整，有些随意，有些口语化
- **自然融入**：
  - 金句要自然融入段落，不要单独成段
  - 可以用更口语化的表达：「说白了就是...」
  - 如果金句很重要，可以在结尾呼应，但不要完全重复

**示例对比**：
```
❌ AI味（金句密度过高）：
**工具是表，方向是里**。AI 是杠杆，但你的**提问方式才是支点**。

掌握了正确的协作方式，AI 就能成为你的**10 倍效率放大器**。

但记住：**真正的 AI 高手，不是知道参数的人，而是能指挥千军万马的人**。

✅ 人味（金句自然出现）：
工具只是外在，提问和思路才是核心。

AI 是放大器，但真正决定效率的，是你「怎么问」和「要去哪里」。

我见过太多人，把 AI 当「实习生」用，给个模糊指令，然后抱怨「它怎么这么笨」。其实问题不在 AI，在我们自己。
```

## 五、过渡层面：自然 > 套路

### ❌ AI味表现
- 过渡句太像模板，缺乏真实感
- 每段结尾都有总结句

### ✅ 人味写法
- **用更自然的过渡**：
  - 「今天这篇文章，就是帮你从第一层跳到第三层」→ 「说回正题...」
  - 「掌握了正确的协作方式，AI 就能成为你的**10 倍效率放大器**」→ 「其实这事儿我也踩过坑...」
- **减少套路化总结**：不是每段都要总结，让读者自己思考

**示例对比**：
```
❌ AI味：
**今天这篇文章，就是帮你从第一层跳到第三层**。

掌握了正确的协作方式，AI 就能成为你的**10 倍效率放大器**。

✅ 人味：
说回正题，怎么从第一层跳到第三层？

其实这事儿我也踩过坑。刚开始用 Cursor 的时候，我也觉得它「不够聪明」，后来才发现，问题不在 AI，在我自己。
```

## 六、类比层面：变化 > 重复

### ❌ AI味表现
- 「踢球」类比出现 3 次，缺乏变化
- 类比过于完美，缺少真实感

### ✅ 人味写法
- **换用不同类比**：不要一直用同一个类比
- **增加真实感**：类比要贴近真实场景

**示例对比**：
```
❌ AI味（重复使用）：
就像踢球，你不能只告诉队友「传球」，得说「传给左边那个穿 10 号球衣的，他正在跑位」。

**类比**：就像踢球，你不能只告诉队友「传球」...

和 AI 协作，就像和队友踢球...

✅ 人味（变化使用）：
就像踢球，你不能只告诉队友「传球」，得说「传给左边那个穿 10 号球衣的，他正在跑位」。

（后面换用其他类比，比如「就像点外卖，你不能只说『来份饭』，得说清楚要什么菜、什么口味、送到哪里」）
```

## 七、格式化层面：适度 > 过度

### ❌ AI味表现
- ❌/✅ 对比过多（几乎每个技巧都有）
- 加粗过多，影响阅读节奏
- **列表过多**：全文有10+个列表，显得机械化
- 每个章节都有列表，显得模板化
- 列表内容过于规整，缺乏口语化表达

### ✅ 人味写法
- **减少格式化标记**：不是每个技巧都要 ❌/✅ 对比
- **适度使用加粗**：只加粗真正重要的句子
- **减少列表数量**：
  - **建议全文列表数量 ≤ 5个**
  - 部分列表改成段落叙述
  - 列表内容更口语化，不要像说明书
  - 不是每个章节都要列表

**示例对比**：
```
❌ AI味（过度格式化）：
**❌ 低效方式**：
- 「添加一个按钮」
- 「让它能展开收起」
- 「样式要好看」

**✅ 高效方式**：
```
在人群黑白名单标题右侧添加展开/收起按钮...
```

**核心原则**：**一次性提供完整上下文**。

✅ 人味（适度格式化）：
我刚开始也犯过这个错。有次让 Cursor 加个按钮，我只说了一句「添加一个按钮」，结果它给我生成了 5 个版本，每个都不一样。

后来我才明白，是我没说清楚位置、样式、功能。

正确的做法是：一次性把话说清楚。比如「在人群黑白名单标题右侧添加展开/收起按钮，用于控制黑名单和白名单内容的显示隐藏。默认展开状态，按钮显示当前状态，图标随状态变化（向上/向下箭头）。参考设计稿：https://mastergo.com/...」
```

## 八、情感层面：真实 > 完美

### ❌ AI味表现
- 语调过于平稳，缺少情绪起伏
- 没有真实的挫败感、惊喜感
- 缺少「我也犯过这个错」的真实反应

### ✅ 人味写法
- **加入真实情绪**：
  - 「我当时也懵了...」
  - 「后来发现原来是...」
  - 「说实话，我也踩过这个坑...」
- **允许不完美的表达**：
  - 偶尔用口语化的感叹（「卧槽」「真的」「太坑了」）
  - 偶尔用不完美的句子

**示例对比**：
```
❌ AI味（过于平稳）：
刚开始用 AI 编程助手时，我也觉得它「不够聪明」，后来才发现，问题不在 AI，在我自己。

✅ 人味（有情绪起伏）：
刚开始用 Cursor 的时候，我也觉得它「不够聪明」。

有次写个按钮，来回改了 5 遍，我当时就火了：「这 AI 是不是有问题？」

后来才发现，问题不在 AI，在我自己。是我没说清楚位置、样式、功能。

说实话，我也踩过这个坑。
```

## 九、留白层面：思考 > 总结

### ❌ AI味表现
- 每个章节都有「核心原则」「底层逻辑」「为什么有效」等总结
- 过于完整，读者没有思考空间
- 附录部分显得多余
- **「写在最后」过于套路**：先总结、再升华、最后互动，完全是模板
- 「掌握了正确的协作方式，AI就能成为你的效率放大器」这种话太官方

### ✅ 人味写法
- **适当留白**：不是每个章节都要总结，让读者自己思考
- **减少过度总结**：核心观点说清楚就行，不需要反复强调
- **简化附录**：如果前面已经说清楚，附录可以简化或删除
- **优化结尾**：
  - **禁止使用「写在最后」这种套路标题**
  - 用具体场景结尾，而不是总结升华
  - 减少说教感，增加真实感
  - 用对话或故事替代总结

## 十、内容重复检查（AI生成典型特征）

### ❌ AI味表现
- **同一案例重复出现**：同一个故事或例子在文章中多次出现
- **金句重复**：同一句金句在文章中出现 2 次以上
- **类比重复**：同一个类比（如「踢球」）在文章中出现 3 次以上
- **表述重复**：相同的表述在不同章节重复出现

### ✅ 人味写法
- **每个案例只用一次**：如果需要在不同章节说明，用不同的案例或角度
- **金句最多出现 1 次**：如果金句很重要，可以在结尾呼应，但不要完全重复
- **换用不同类比**：不要一直用同一个类比，可以换用其他生活场景
- **检查重复表述**：写完文章后，搜索关键词，确保没有重复的段落

**示例对比**：
```
❌ AI味（重复案例）：
### 需求层：说清楚要什么
我刚开始也犯过这个错。有次让 Cursor 加个按钮，我只说了一句「添加一个按钮」，结果它给我生成了 5 个版本，每个都不一样。

### 1. 需求描述：一次性说清楚，别让 AI 猜
我刚开始也犯过这个错。有次让 Cursor 加个按钮，我只说了一句「添加一个按钮」，结果它给我生成了 5 个版本，每个都不一样。

✅ 人味（不同案例）：
### 需求层：说清楚要什么
我刚开始也犯过这个错。有次让 Cursor 加个按钮，我只说了一句「添加一个按钮」，结果它给我生成了 5 个版本，每个都不一样。

### 1. 需求描述：一次性说清楚，别让 AI 猜
还有一次，我想让 AI 优化一段代码，只说了「优化一下」，结果它给我改得面目全非，我花了 2 小时才改回来。
```

## 十一、数字使用检查（避免过于工整）

### ❌ AI味表现
- 「10 个坑」「10 个技巧」「10 倍效率」这种数字过于工整
- 所有数字都是整数，缺乏真实感

### ✅ 人味写法
- **用具体数字替代整数**：
  - 「10 个坑」→ 「这 10 个坑」（如果确实是 10 个，可以保留，但要确保真实）
  - 「10 倍效率」→ 「从 2 小时缩短到 15 分钟」（用具体数据替代倍数）
- **允许不完美的数字**：
  - 「我观察了 20+ 个程序员」
  - 「大概用了 3-4 个小时」

## 十二、检查清单：写完后的自检

### 必须检查项（核心）

**【新增】洞见与论证质量**
- [ ] **洞见唯一性检查**：文章是否有一个具体的核心洞见？用一句话写出来——「读完这篇文章，读者会知道一件以前不知道的事：______」。如果这句话是一个「大家都知道的常识」，说明洞见不够深，必须返工。
- [ ] **论证完整性检查**：文章中每个核心论点，是否都有「案例 + 原理 + 反例/边界」三要素支撑？还是只有结论？只有结论不是论证，是断言。
- [ ] **反直觉设计检查**：文章至少有一处「大家以为是X，其实是Y」的反转。有吗？在哪里？如果没有，文章缺少让人「哦！」的时刻。
- [ ] **深度层级检查**：把文章的核心观点提炼为一句话——它是「描述现象」（第1-2层）还是「解释机制」（第4层）？描述现象不是洞见。参考 `深度论证框架.md` 的 5 层论证法检查。
- [ ] **边界说明检查**：文章的核心结论是否说明了「适用边界」？在什么情况下成立，什么情况下不适用？没有边界的观点，一个反例就能打倒。

**【原有】人味与表达质量**
- [ ] **结构编号检查**：是否还有「一、二、三」编号？是否改用了描述性标题或故事化标题？
- [ ] **句式重复检查**：「后来我明白了/学会了」出现几次？（建议≤1次）是否换用了不同表达？
- [ ] **列表数量检查**：全文列表数量是否≤5个？是否将部分列表改成了段落叙述？
- [ ] **内容重复检查**：搜索关键词，确保没有重复的案例、金句、类比
- [ ] **符号标记检查**：是否减少了格式化标记（❌/✅、加粗）？每个技巧都用符号了吗？
- [ ] **真实细节检查**：是否增加了真实细节（具体场景、数据、对话）？是否有「@老王」「昨天在『AI编程实战群』」这种具体信息？案例是否有前因后果和情绪？
- [ ] **金句密度检查**：全文金句数量是否≤2句？金句是否自然融入段落，而不是单独成段？
- [ ] **口语化检查**：是否增加了口语化表达？是否减少了「信息论视角」「底层逻辑」等术语？是否将正式表达改成了口语化？
- [ ] **结构变化检查**：是否打破了固定结构，增加了变化？是否有些章节用故事，有些用对话？章节长度是否有变化？
- [ ] **类比变化检查**：是否换用了不同类比，避免重复？同一个类比是否出现 3 次以上？
- [ ] **情绪检查**：是否加入了真实情绪和反应？是否有「我当时就火了」「说实话，我也踩过这个坑」这种表达？
- [ ] **结尾检查**：是否还有「写在最后」这种套路标题？结尾是否用具体场景替代了总结升华？

### 重要检查项
- [ ] **过渡句检查**：是否减少了套路化过渡句？是否用「说回正题」「其实这事儿我也踩过坑」替代了「今天这篇文章，就是帮你...」？
- [ ] **留白检查**：是否适当留白，减少过度总结？不是每个章节都要总结
- [ ] **数字检查**：是否避免了过于工整的数字？是否用具体数据替代了「10 倍效率」这种表述？
- [ ] **节奏检查**：章节长度是否有变化？是否有些章节更短，有些更长，制造节奏感？

### 可选优化项
- [ ] 是否简化了附录部分？
- [ ] 是否允许了不完美的句子？
- [ ] 结尾是否过于模板化（「今日互动」「推荐阅读」「关键词」）？

---

## 核心原则总结

1. **洞见 > 汇总**：文章要有让读者「哦！」的核心洞见，不是大众观点的整合
2. **机制 > 现象**：解释「为什么」比描述「是什么」更有价值
3. **论证 > 断言**：每个核心观点都要有案例+原理+边界支撑
4. **真实 > 完美**：允许不完美的表达，增加真实感
5. **具体 > 抽象**：补充具体细节，增加个人色彩（场景、数据、对话、情绪）
6. **自然 > 刻意**：让金句和过渡自然出现，不要套路化（金句≤2句，自然融入）
7. **变化 > 重复**：打破固定格式，增加结构变化（不用编号、不同开头方式、节奏变化）
8. **口语 > 书面**：用口语化表达，减少术语堆砌（避免句式重复）
9. **适度 > 过度**：减少格式化标记，增加段落叙述（列表≤5个）
10. **留白 > 总结**：适当留白，让读者自己思考（去掉「写在最后」）

---

**记住**：读者想看到的是「一个真实的人，分享了一个我以前从没想过的角度」。**好文章 = 真实的人 + 真实的洞见 + 完整的论证 + 有人味的表达**。四个条件，缺一不可。
