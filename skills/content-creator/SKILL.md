# 内容创作者 (Content Creator)

## 概述

本 Skill 提供一套标准化工作流，基于「AI 编程实战」方法论，从人设、结构、深度拆解到最终封装，全链路支持**高质量内容生产**。

**两条铁律，缺一不可：**

1. **洞见铁律**：每篇文章必须有一个让人「哦，原来如此」的核心洞见。不是整合大众观点，是穿透本质的那一刀。文章没有洞见，写再多也只是热闹。
2. **人味铁律**：真实 > 完美。所有内容必须经过「去AI味」检查，读者感觉到的是一个真实的人在分享经验，而不是 AI 在输出模板。

---

## 工作流决策树

分析输入类型，选择对应工作流：

- **录音/转录稿** → 工作流 A（录音转文章）
- **想法/热点/一句话** → 工作流 B（灵感孵化）
- **已有文章/初稿** → 工作流 C（文章改编/润色）
- **文章转视频** → 工作流 D（视频脚本）
- **文章转小红书** → 工作流 F（小红书矩阵）
- **竞品文章拆解** → 工作流 E（深度拆解）

所有**生成类**工作流（A/B/C）结束后，必须执行 **Package Phase**（封装）。

## 工作流 A：录音转文章

**目标**：将口语素材转化为**2000字左右、有核心洞见、逻辑严密**的专业文章。

### 阶段一：清洗（Clean）

去除口语冗余、重复、语气词。提炼出原始素材中的**核心观点句**（每个观点用一句话写下来，为下一阶段准备）。

**输出**：`流畅版.md` + 一份「核心观点提炼清单」。

### 阶段二：深度重构（Deep Structure）

**这是文章质量的决定性阶段。**

1. **洞见挖掘**（必须先做）：
   - 从「核心观点提炼清单」中找到最反常识、最有价值的那一个。
   - 问自己：「如果读者没看这篇文章，他们能想到这个结论吗？」如果答案是「能」，说明洞见不够，需要继续挖掘。
   - 参考 `references/深度论证框架.md` 的「5层深度论证法」，确认洞见深度。

2. **结构选型**：引用 `references/文章结构.md`，选择最能承载这个洞见的结构模式。

3. **饱和式论证**：
   - 每个核心观点必须用「论点 → 案例 → 原理 → 反例 → 结论」五步完整论证。
   - 若素材不足，根据人设和专业知识进行逻辑补全，**但所有补充内容必须真实可信**，不编造数据。
   - 文章必须包含至少一处「颠覆直觉」的分析（读者看到这里会停下来，重新读一遍）。

4. **文章节奏设计**：按照「开场高能 → 中段转折 → 收尾升华」安排能量起伏，避免平铺直叙。

### 阶段三：洞见质检（Insight QA）⭐️

**这是新增的质量关卡，防止「结构正确但内容平庸」。**

对文章进行以下逐项检查：

- [ ] **洞见唯一性**：文章是否有一个让人「哦！」的核心洞见？这个洞见是否贯穿全文？
- [ ] **论证完整性**：每个核心论点是否都有案例+原理+反例的支撑？还是只有结论？
- [ ] **反直觉设计**：文章至少有一处「大家以为是X，其实是Y」的反转。有吗？在哪里？
- [ ] **深度检验**：把文章的核心观点提炼成一句话，这句话是「描述现象」还是「解释机制」？后者才是真深度。
- [ ] **读者收获感**：读完这篇文章，读者能得到一个「以前不知道、现在知道了」的认知升级吗？

如果任何一项回答是「否」，必须回到阶段二继续深化，不能跳过。

### 阶段四：润色（Refine）

- 引用 `references/人设.md` 和 `references/金句.md` 确认风格调性。
- 应用「反爹味」原则，加入类比和口语化表达。
- **执行完整去AI味检查**：引用 `references/去AI味指南.md`，检查全部 14 个核心检查项（含新增的洞见真实性和论证完整性检查）。
- 发现问题，立即修复，修复后再输出。

**输出**：`公众号文章_final.md`。

## 工作流 B：灵感孵化

**目标**：从一个点子扩展为一篇**2000字左右的深度长文**，让读者看完直呼「牛」。

### 步骤一：破局点挖掘（Insight Mining）

**这是决定文章天花板的一步，不能跳过。**

对输入的想法，连续追问以下问题：

1. 「这件事大家普遍的认知是什么？」（定位共识）
2. 「这个共识哪里错了，或者不完整？」（找到裂缝）
3. 「裂缝背后，更本质的问题是什么？」（挖到底层矛盾）
4. 「这个底层矛盾，用什么框架可以解释清楚？」（建立分析模型）
5. 「如果用这个框架去看，能得出什么反直觉的结论？」（形成洞见）

每一层都要写出来。第5层的答案，就是文章的核心洞见。

还可以尝试以下切入角度，选最能制造惊喜的那个：
- 「反常识」角度：大家觉得是A，其实是B
- 「根因」角度：大家在解决症状，真正的病根在这里
- 「第一性原理」角度：从最基础的事实推导，推翻所有中间假设
- 「长期视角」角度：短期看X，长期看Y，绝大多数人只看到了X

### 步骤二：论证骨架（Argument Framework）

洞见确定后，搭建完整论证骨架：
- 核心论点（一句话）
- 支撑论点 1 → 证据 → 反例
- 支撑论点 2 → 证据 → 反例
- 支撑论点 3 → 证据 → 反例
- 边界条件（这个结论在什么情况下不成立）
- 结论升华

参考 `references/深度论证框架.md` 确保论证质量。

### 步骤三：深度起草（Deep Draft）

引用 `references/文章结构.md` 进行饱和式扩写：
- 每个论点必须有具体案例或数据支撑（不能只有结论）。
- 必须包含至少一处反直觉分析。
- 字数目标：2000字左右（靠逻辑撑起来，不靠堆列表）。
- 文章节奏：开场要抓人，中段要有转折，结尾要有余味。

### 步骤四：洞见质检 + 润色

同工作流 A 的阶段三 + 阶段四，执行完整的质检和去AI味检查。

## 工作流 C：文章改编

**目标**：旧文新写，让深度和专业度有质的飞跃，不是改几个词，是重新「立骨」。

### 分析阶段

- 提取旧文的核心价值点（有什么是值得保留的？）
- **升维思考**：从更本质的角度（工程化思维、长期主义、第一性原理）重新解读这个话题，能得出比原文更深的结论吗？
- 找到原文「点到为止、没有说透」的地方，这些是改编的重点。

### 重写阶段

- 引用 `references/文章结构.md` 调整结构，增加论证密度。
- 执行洞见质检：改编后的文章是否有真正的洞见？比原文的认知密度高了吗？
- 执行去AI味检查：引用 `references/去AI味指南.md`，检查全部 14 个核心检查项。

## 工作流 D：短视频脚本

**目标**：将文章转化为高完播率的口播脚本；若用户要求「深度」「讲透」「原理向」，必须同时产出**可执行的口播密度**，避免「短但空」。

### 两档产出（默认问清或双轨输出）

| 档位 | 时长参考 | 字数参考（中文口播） | 用途 |
| ---- | -------- | ------------------- | ---- |
| **短平快** | 45–75 秒 | 约 180–350 字 | 信息流、强 Hook、收藏向清单 |
| **深度口播** | 90–150 秒 | 约 450–800 字 | 视频号、关注后观看、建立专业信任 |

> 口播语速按约 **200–240 字/分钟**（清晰、有停顿）估算；含数字、英文缩写处自动放慢半拍。

**约束条件（两档共用）**：

- 开头：第一句必须是 Hook（打破认知/制造焦虑/反常识发问）。
- 语言：口语化优先；专业词**必须带一句人话翻译**（例如「token → 按字块计费、代码也算字」）。
- **禁止**只有「怎么做」没有「为什么」：至少回答读者会在评论区追问的 **1 个机制问题**（见下方「深度三问」）。

**生成结构（深度口播必选）**：

痛点 → 误区揭示 → **机制层（为什么）** → **边界与反例（什么时候不灵）** → 实操建议 → 行动收口。

**分镜**：为每一段标注画面/特效建议，增加制作可行性。

### 深度口播专属：上线前自检（Video Depth QA）

生成或改编短视频稿时，逐项检查；任一项为「否」则补段，不得交短稿充数：

- [ ] **机制**：是否说清楚「现象背后的结构原因」（例如窗口竞争、注意力稀释、中间位置弱势），而不止「要清理」？
- [ ] **数字有锚**：出现百分比、阈值（如「七成」）时，是否解释**为什么是这档**、**不是更低/更高**各会有什么代价？（允许诚实说「经验阈值/产品余量」，禁止编造厂商内部参数。）
- [ ] **反例/边界**：是否有一句「什么情况可以不那么早压」「什么必须立刻 clear」？
- [ ] **信息增量**：删掉所有形容词，脚本是否仍有一个读者愿意转述给同事的点？
- [ ] **去 AI 味**：避免清单腔、避免「一、二、三」标题腔；金句全文 ≤5 句且不单独成段（与 `references/去AI味指南.md` 对齐）。

### 「读者会追问的三件事」模板（至少覆盖 1 件，深度口播建议 2 件）

在稿内用口语自然嵌入，不必单独立标题：

1. **为什么是这档阈值？**（经验值 / 余量留给工具返回与模型长回复 / 避免贴顶后被动截断）
2. **和「更早一点做」比，多等什么会吃亏？**（信号被稀释、纠错成本、时间浪费）
3. **不做的最坏情况是什么？**（胡编接口名、改错文件、上下文打架）

### 与长文工作流的对齐

- 核心洞见引用 **洞见铁律**：短视频可以短，但不能没有「哦，原来如此」的那一刀；若素材来自长文，**不得只缩句**，要保留一条论证链（机制或对比）。
- 论证密度参考 `references/深度论证框架.md` 的「论点 → 案例 → 原理」压缩版（短视频可合并步骤，但不可只剩结论）。

**封装（工作流 D 可选用）**：3 条 ≤80 字朋友圈文案 + 1 条置顶评论「追问答疑」预埋（回应机制向追问）。

## 工作流 F：小红书矩阵

**目标**：将长文章拆解为一系列高传播力的小红书笔记。

1. **拆解评估**：引用 `references/小红书指南.md`，分析知识点密度，决定拆解数量（通常 3-5 篇）。
2. **笔记生成**：
   - 标题：每篇 5 个备选（含 emoji 和情绪词）。
   - 正文：Hook + 干货 + 互动，emoji 含量高，一屏可读完。
   - 首图建议：详细描述视觉元素。
   - 标签：生成精准 tag。

---

## 工作流 E：深度拆解

**目标**：向优秀竞品文章学习思维模型，提炼可复用的写作策略。

执行分析：引用 `references/深度拆解模型.md`，输出 TL;DR、文章地图、观点-证据表、写作技巧分析、可复用模型提炼。

---

## 通用封装阶段（Package Phase）

*适用于 A、B、C 工作流的最后一步。*

引用 `references/标题与配图指南.md`，输出：
1. **5 个爆款标题**（痛点型/反常识型/方法型/清单型/强对比型），优先用「」。
2. **配图方案**（首图 + 内文图建议）。
3. **3 条朋友圈文案**（≤80字，口语化、利他、去爹味）。
