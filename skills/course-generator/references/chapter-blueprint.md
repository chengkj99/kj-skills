# 章节结构蓝图（chapter-blueprint）

> 一篇课程章节怎么排，才能让学员「读得进、学得会、用得上」。

## 通用底层逻辑

主流付费课的章节都暗含同一条认知动线，照着走就不会散：

```
为什么要学（痛点 / 场景）→ 是什么（概念，最短路径讲清）
→ 怎么用（操作 / 示例，可复现）→ 怎么用好（坑 / 边界 / 我的建议）
→ 你能带走什么（小结 / 可复用清单）
```

四条通用要求：

1. **每章开头给「学完你能干什么」**，一句话，别铺垫。学员要先知道这趟值不值。
2. **概念服务于操作**。不为讲概念而讲概念——讲完一个概念，紧跟着就要落到「所以你可以这样做」。
3. **至少一个学员能跟着复现的东西**：一段可运行代码、一组可照抄的配置、一个能照做的步骤序列。光看懂不算会。
4. **结尾留「可带走物」**：一张对比表 / 一份清单 / 一个模板 / 一句能记住的判断。

## 类型 A：概念讲解型

讲清一个原理或模式（如「Agent 模式详解」「MCP 是什么」）。

```
★ 开篇：用一个痛点或场景切入，点出「这一讲要把 X 讲透」
★ 这东西到底是什么：最短路径 + 一个日常类比（如 MCP = USB-C 接口）
   它能干什么 / 不能干什么：直接列能力边界，别含糊
★ 一个完整示例：演示它在真实场景里怎么跑（哪怕是文字模拟一次工作流）
   和相近概念的区别：用对比表格（如 Ask / Edit / Agent 三模式对比）
★ 常见的坑 + 我的建议：新手最容易栽哪、什么时候用什么时候不用
★ 小结：一句话把这一讲钉死
```

要点：概念型最怕「讲了一堆名词，学员还是不知道拿它干嘛」。**每个概念后面都要接一句「所以你能……」**。

## 类型 B：工具操作型

教会一个配置或功能（如「配置 MCP 服务器」「接入国产模型」）。

```
★ 开篇：这个功能解决什么问题（不解决就先讲清痛点）
   前置条件：需要装什么 / 提前准备什么（如「需要先装 Node，第 03 讲讲过」）
★ 分步操作：编号步骤，每步「做什么 + 怎么做 + 看到什么算成功」
   配置 / 代码块：完整可照抄，关键行加注释
★ 验证：怎么确认配好了 / 跑通了（给一个明确的验证动作和预期结果）
   排错：最常见的 2–3 个失败原因 + 解决办法
★ 进阶 / 推荐：常用的几个选项或扩展（点到为止，不堆砌）
★ 小结：关键步骤回顾 / 一张速查表
```

要点：操作型的命脉是**可复现**。步骤不能跳，路径要写全（如配置文件具体路径），「成功长什么样」必须明说，否则学员卡住不知道是不是自己错了。

## 类型 C：实战项目型

带学员做出一个完整产物（如「图片压缩桌面应用-核心功能实现」）。常常拆成多讲（需求 → 搭建 → 核心功能 → 打包发布），本技能一次写其中一讲。

```
★ 开篇：这一讲要做出什么（最好有「做完长这样」的描述或截图位）
   承上：上一讲做到哪了，这一讲在整体里的位置（多讲项目必写）
★ 本讲目标拆解：把这一讲的活拆成 2–4 个可验收的小任务
★ 动手做：按小任务推进，每个任务 = 需求说清 → 给 AI 的提示词 → 产出代码 → 验证
   关键代码讲解：不是逐行念，是讲「为什么这么写 / 这里容易错」
★ 阶段验收：这一讲做完，怎么确认成果是对的（运行起来 / 功能可用）
   踩坑提示：这一步真实会遇到的问题
★ 小结 + 预告：本讲产出 + 下一讲要做什么
```

要点：实战型要让学员**跟着做能跑通**。给 AI 的提示词要真实可用（体现第 11 讲那种「目标 / 边界 / 示例 / 验收」结构），代码要完整不省略，别出现「这里省略若干代码」。

## 类型 D：方法论型

讲思维方式或工作流（如「写出高质量 Prompt」「我对 AI 编程的思考」）。

```
★ 开篇：一个反常识结论或扎心痛点（如「大多数人 AI 用不好，不是工具问题，是 Prompt 问题」）
★ 核心模型 / 框架：把方法论提炼成一个可记忆的结构（如 Prompt 四要素）
★ 正反对比：模糊做法 vs 工程化做法，用具体例子，差距要看得见
★ 拆解每个要素：逐条讲怎么落地，配小例子
   常见误区：大家通常怎么做错的
★ 一套可复用的清单 / 模板：学员能直接拿去用
★ 小结：把方法论收成一两句能背下来的话
```

要点：方法论最怕空。**每个观点都要落到可执行**——给了判断就要给清单 / 模板 / 步骤，让感悟变成能照做的东西。

## 开篇怎么写

**绝对不用定场白**（不写「大家好，我是康健」「欢迎来到本讲」）。文字课程直接进正题，开篇三选一：

1. **痛点切入**：「先说一个残酷的真相：大多数人 AI 用不好，不是工具问题，是 Prompt 的问题。」
2. **场景切入**：「你在一个已有 sort.js 的项目里，对 Agent 说：帮我加单元测试……」
3. **反常识结论**：「首先我要泼盆冷水：编程的本质没有变。」

开篇之后第一段内，让学员知道**这一讲要解决什么、学完能干什么**。

## 收尾怎么写

- 用「可带走物」收尾：小结 / 清单 / 速查表 / 一句钉死的判断。
- 多讲项目要带**预告**：下一讲做什么，给个钩子。
- **署名**：可在全章最末用一句个人化的话收束（口吻像康健本人的旁白），不强制；开头一律不署名。是否署名跟随用户偏好，没说就默认不加，保持干净。

## 视觉元素指南

> 课程要图文并茂，但图不是装饰——只在「一张图能省掉一段解释」的地方放。

### 核心原则

- 视觉服务于理解，不是填充。问自己：**「这里没有图，学员会卡住吗？」** 会，就放；不会，就省。
- 每章至少有 1 处非代码块的视觉（截图 / 流程图 / 对比图 / 录屏建议）。
- 代码块直接写进正文；截图 / 流程图 / 录屏无法由 AI 生成，用**占位标记**精确描述，供用户后续补充。

### 占位标记格式（必须统一）

```
<!-- [截图] 描述：拍什么界面 + 什么状态 + 重点标注什么 -->
<!-- [流程图] 描述：画什么逻辑 + 几个节点 + 核心是什么 -->
<!-- [录屏建议] 描述：录哪段操作 + 从哪步到哪步 + 时长估计 -->
<!-- [对比图] 描述：左边是什么（错误/旧） + 右边是什么（正确/新） -->
```

描述要具体到「不需要猜，看标记就知道拍什么」。举例：

- ✅ `<!-- [截图] claude --version 命令运行成功后的终端输出，重点：版本号那一行 -->`
- ❌ `<!-- [截图] 安装成功的截图 -->`（太模糊，拍出来可能对不上）

### 四类章节推荐视觉位置

| 章节类型 | 主力视觉类型 | 推荐放置时机 |
|---|---|---|
| A 概念讲解型 | 流程图（工作原理）、对比图（横向对比） | 讲完原理后紧跟流程图；对比表格旁加对比图 |
| B 工具操作型 | 截图（每步关键界面状态）、录屏建议 | 每个关键操作步骤后放截图；整章末尾放录屏建议 |
| C 实战项目型 | 截图（阶段成果）、录屏建议、代码块 | 每个小任务完成后放成果截图；最终验收前放录屏建议 |
| D 方法论型 | 流程图（框架/模型图）、对比图（好坏做法） | 核心模型提出后放流程图；正反对比段落配对比图 |

### 代码块写作要求

代码块不算视觉占位，是正文的一部分，标准更严：

- 完整可运行，不出现「省略若干代码」。
- 关键行加注释，说明「为什么这么写 / 这里容易踩坑」。
- 指定语言（` ```bash` / ` ```json` / ` ```javascript`），方便高亮渲染。
- 命令类代码：注释用 `#`，说明每条命令的作用及预期输出。
