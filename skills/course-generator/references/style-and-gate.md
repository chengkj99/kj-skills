## 课程章节的写作风格

### 语气

- **同行，不是老师**。平视学员，「我们来看看」「你会发现」，不是「你应该」「你必须」。
- **工程视角**。讲收益也讲代价、讲能力也讲边界、讲怎么用也讲什么时候别用。不吹工具万能。
- **坦诚**。该泼冷水泼冷水（「编程的本质没变」「这个回滚和 Git 不是一回事」），不画大饼。

### 说人话

- 抽象概念**必须**配一个日常类比：MCP = USB-C 接口；写代码 = 搬砖盖房 → 当建筑设计师；模糊需求 = 给实习生派活不说清。
- 类比要**贴切且常见**，不要为了类比硬凑。一个概念一个好类比就够，别堆。
- 长句拆短。一句话讲一件事。

### 强调与排版

- **关键判断句加粗**，但克制：一个小节 1–2 处，加粗滥用等于没加粗。
- 善用结构化呈现，但**按内容选形式**，不要为了好看硬套：
  - 对比 / 选型 → 表格（如模式对比、模糊版 vs 工程化版）。
  - 操作流程 → 编号步骤。
  - 配置 / 示例 → 代码块，关键行加注释，能跑通。
  - 并列要点 → 短列表。
- 适当用「>」引用块放一句关键提醒或可截图的判断。

### 接地气的案例

- 案例要**真实、具体、有项目质感**：给文件名（`src/components/UserList.vue`）、给真实命令（`npm run test`）、给会真实遇到的报错。
- 不用悬浮的「假设我们有一个系统……」。要么用课程贯穿的实战项目，要么用程序员日常真碰到的场景。

## 禁用词与替代说法

这些词生硬、或有 AI 腔、或增加理解成本，**禁止使用**，用右边的说法替代：

| 禁用 | 改成 |
|------|------|
| 铁律、金科玉律 | 这条最重要 / 记住一点 / 别破这个底线 |
| 赋能、抓手、闭环、心智 | 直接说具体在帮什么 / 让你能做什么 |
| 综上所述、总而言之 | 小结 / 说到底 / 一句话 |
| 接下来让我们一起 | 我们来看 / 下面 |
| 在当今 XX 时代、随着 XX 的发展 | 直接讲事，别铺垫宏大背景 |
| 值得注意的是、需要指出的是 | 注意 / 这里有个坑 |
| 赋予、助力、打造、构建（滥用时） | 做 / 写 / 搭 / 让它能 |

开篇绝不出现：「大家好」「我是康健」「欢迎来到」「本讲将介绍」。

## 质量门禁清单

交付前**逐条**核对，任一不过就改到过为止。

### 一、去 AI 味

- [ ] 开篇不是定场白 / 不是「在 XX 时代」式空铺垫，是痛点 / 场景 / 反常识结论。
- [ ] 没有「禁用词」表里的词。
- [ ] 没有「首先…其次…最后…」这种三段式空壳；没有每段都工整对仗的 AI 腔。
- [ ] 没有正确的废话（删掉也不影响理解的句子）。
- [ ] 句子长短有节奏，不是一水儿的长复合句。

### 二、康健味在场（理性 + 人味都要有）

- [ ] **理性**：有模型 / 清单 / 对比 / 边界判断，观点能落地。
- [ ] **人味**：有「我」的视角、真实经验或踩过的坑、平视语气，不爹味、不说教。
- [ ] 至少有 1 处能截图记住的判断句。
- [ ] 专业准确：技术细节、命令、路径、概念没硬伤（拿不准的地方标注，不编造）。

### 三、课程有效性（这才是课程，不是文章）

- [ ] 开头让学员知道「学完能干什么」。
- [ ] 至少有一个学员能**复现**的东西（可跑代码 / 可抄配置 / 可照做步骤）。
- [ ] 操作型：每步有「成功长什么样」；实战型：代码完整不省略、有阶段验收。
- [ ] 讲了概念也讲了「所以能拿它干嘛」，没有为讲而讲。
- [ ] 结尾有「可带走物」（小结 / 清单 / 表 / 模板）。

### 四、材料忠实

- [ ] 内容基于参考材料，没有凭空编造事实 / API / 配置。
- [ ] 参考材料没覆盖到、又必须讲的地方，要么基于通用可靠知识保守处理，要么标注「建议核实 / 补充材料」，不硬编。

### 五、图文并茂

- [ ] 每章至少有 **1 处非代码块的视觉占位标记**（截图 / 流程图 / 录屏建议 / 对比图）。
- [ ] 占位标记格式统一（`<!-- [类型] 描述 -->`），描述具体到「看标记就知道拍什么」，不能是「XX 截图」这种模糊描述。
- [ ] 代码块：完整可运行、指定语言、关键行有注释，不出现「此处省略」。
- [ ] 视觉放置位置有意义——紧跟「最需要图辅助理解」的段落，不是随机插在节与节之间。
