# Skills 仓库健康度最佳实践

## 参考模型

用三个方向校准判断：

- `anthropics/skills`：官方规范感。重点学习自包含 skill 目录、`SKILL.md` 入口、模板、规范和测试。
- `addyosmani/agent-skills`：工程流程感。重点学习生命周期拆分、质量门禁、验证证据和反偷懒规则。
- `wshobson/agents`：生态治理感。重点学习多平台分发、质量评分、跨 harness 可移植性和文档体系。

不要机械模仿目录名。提炼标准后，按当前仓库的目标用户和平台落地。

## 仓库级必要条件

1. 有稳定结构：`skills/<skill-id>/SKILL.md` 是主路径，`references/`、`scripts/`、`examples/`、`assets/` 只在有真实用途时出现。
2. 有清晰索引：README 或等价入口能让人快速知道有哪些 skill、适合什么场景、如何安装或同步。
3. 有贡献规则：新增、重命名、删除 skill 时，要同步用户索引、插件 manifest、平台安装文档、帮助命令和 sync 说明。
4. 有验证路径：至少能检查 frontmatter、目录命名、注册可见性；复杂 skill 要有样例、测试或 dry-run。
5. 有去重机制：定期判断 skill 之间是完全重复、强重叠、互补，还是流水线依赖。
6. 有安全边界：凡是写文件、提交代码、发布内容、调用外部服务、读私有知识库的 skill，都要说明权限和禁区。

## 单个 Skill 质量标准

### Frontmatter

- `name` 与目录名一致。
- `description` 同时说明“做什么”和“什么时候用”。
- `description` 包含足够触发词，但不塞完整工作流。
- 不保留 TODO、模板文案或泛泛的 “helps with X”。

### Body

- 开头说明目标和适用场景。
- 流程是命令式，可执行，不只是价值宣言。
- 长背景进入 `references/`，机械动作进入 `scripts/`。
- 输出格式、文件路径、验收标准写清楚。
- 对容易误用的场景写 Red Flags 或不适用说明。

### Bundled Resources

- `references/`：放详细标准、领域知识、长模板。超过 100 行时最好有目录或清晰标题。
- `scripts/`：放重复、确定性、易出错的动作。脚本必须可直接运行，优先标准库和少依赖。
- `assets/`：放最终产物要使用的模板、图片、字体、样例工程，不当作说明文档仓库。

## 评分建议

每项 0-10 分，最后求和：

| 维度 | 高分表现 | 低分信号 |
|---|---|---|
| 结构一致性 | 目录、入口、资源清晰 | 文件散落、入口不明 |
| 触发准确性 | `description` 可路由 | 描述泛、误触发多 |
| 职责边界 | 一个 skill 一个核心任务 | 万能 skill、阶段混杂 |
| 渐进披露 | 主文件精简，按需读引用 | `SKILL.md` 过长 |
| 可验证性 | 有命令、证据、验收 | 只说“确保质量” |
| 输出契约 | 格式稳定可复用 | 输出随意 |
| 可维护性 | 命名清楚，脚本靠近能力 | 重复逻辑、隐性依赖 |
| 可发现性 | README/manifest/help 同步 | skill 存在但找不到 |
| 去重治理 | 关系分类明确 | 多个 skill 抢同一任务 |
| 安全边界 | 副作用明确受控 | 写入/发布/提交无约束 |

评级：

```text
90-100：标杆级
80-89：生产可用
70-79：可用但需要治理
60-69：脆弱，需要先清理
<60：不建议继续扩张
```

## 重复关系分类

- 完全重复：目标用户、输入、流程、输出几乎相同。建议合并。
- 强重叠：都覆盖同一主任务，但某个 skill 更完整。建议保留主 skill，另一个改名、收缩或并入引用。
- 互补：服务同一大目标的不同阶段。建议写清上下游关系。
- 流水线依赖：一个 skill 产出另一个 skill 的输入。建议在 README 协作关系中体现。

判断重复时不要只看名字，要读 `SKILL.md` 的 frontmatter、流程和输出契约。

## 高质量审计输出

好的健康度报告应该：

- 先给总评和分数，而不是堆扫描日志。
- P0/P1/P2 分级，说明影响和修法。
- 引用具体路径。
- 区分自动扫描发现和人工判断。
- 给出 30/60/90 分钟可以执行的迭代计划。
- 明确哪些问题本轮不建议改，避免越审越散。
